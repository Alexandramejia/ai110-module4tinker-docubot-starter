# DocuBot Model Card

This model card is a short reflection on your DocuBot system. Fill it out after you have implemented retrieval and experimented with all three modes:

1. Naive LLM over full docs  
2. Retrieval only  
3. RAG (retrieval plus LLM)

Use clear, honest descriptions. It is fine if your system is imperfect.

---

## 1. System Overview

**What is DocuBot trying to do?**  
Describe the overall goal in 2 to 3 sentences.

> DocuBot answers developer questions about a sample app using only its own docs. It compares three modes: no retrieval, retrieval only, and retrieval plus an LLM (RAG). This shows how grounding affects answer accuracy.

**What inputs does DocuBot take?**  
For example: user question, docs in folder, environment variables.

> A question typed at the CLI or a sample query from dataset.py. The .md and .txt files in docs/. A GEMINI_API_KEY when an LLM mode is used.

**What outputs does DocuBot produce?**

> Naive mode gives a free form answer with no sources cited. Retrieval only gives raw snippets and filenames with no synthesis. RAG gives an answer built from the retrieved snippets, citing the file it used, or a refusal saying "I do not know based on the docs I have."

---

## 2. Retrieval Design

**How does your retrieval system work?**  
Describe your choices for indexing and scoring.

- How do you turn documents into an index?
- How do you score relevance for a query?
- How do you choose top snippets?

> Each doc is tokenized into lowercase words and used to build an inverted index mapping each word to the files that contain it. Documents are also split into paragraphs so answers come from a specific snippet rather than a whole file. A paragraph is scored by counting how many of the query's meaningful words (stopwords removed) appear in it. A paragraph only counts as evidence if at least half those words appear in it. The top scoring paragraphs (default 3) are returned.

**What tradeoffs did you make?**  
For example: speed vs precision, simplicity vs accuracy.

> The system favors simplicity and safety over recall. Whole word matching avoids false positives from substrings. But it also means singular and plural forms like "user" and "users" do not match each other. Identifiers with underscores like user_id split into pieces that do not match plain words like "table" or "field." Paragraph chunking keeps snippets short and easy to check. But it can also separate a heading from its content and cut a real answer in half. These choices lower the risk of hallucination but can cause the system to miss an answer that is actually in the docs.

## 3. Use of the LLM (Gemini)

**When does DocuBot call the LLM and when does it not?**  
Briefly describe how each mode behaves.

- Naive LLM mode: always calls Gemini with just the question, no docs, no retrieval.
- Retrieval only mode: never calls the LLM, just returns raw snippets from the index.
- RAG mode: retrieves top snippets first, then calls Gemini with only those snippets plus the question.

> Naive mode is pure generation. Retrieval only is pure search. RAG combines both, so the model only ever sees what the retriever chose to hand it.

**What instructions do you give the LLM to keep it grounded?**  
Summarize the rules from your prompt. For example: only use snippets, say "I do not know" when needed, cite files.

> The RAG prompt tells the model to answer using only the snippets. It must not invent functions, endpoints, or config values. It must reply with the exact phrase "I do not know based on the docs I have." when the snippets are not enough. It must name the file it relied on when it does answer.

## 5. Failure Cases and Guardrails

**Describe at least two concrete failure cases you observed.**  
For each one, say:

- What was the question?  
- What did the system do?  
- What should have happened instead?

> Failure case 1: "Which fields are stored in the users table?" Retrieval only and RAG both answered "I do not know," even though DATABASE.md lists the exact columns under a users heading. The cause is whole word matching. The table contains the identifier user_id, which tokenizes to "user" and "id," never the plural word "users." It also never contains the words "field," "table," or "stored." The required word overlap is never met, so the paragraph gets thrown out before it is even scored. It should have found and used that table.

> Failure case 2: "What does the /api/projects/<project_id> route return?" Retrieval only returned just the heading "### GET /api/projects/<project_id>" with none of the response example underneath it. The heading and its content landed in two separate paragraphs after blank line splitting. RAG then correctly refused, since the snippet it got had no real content. The docs answer this question two paragraphs down. Chunking should have kept the heading and its response together.

**When should DocuBot say "I do not know based on the docs I have"?**  
Give at least two specific situations.

> When no paragraph shares enough meaningful words with the question, such as asking about payment processing in docs that never mention payments. Also when the only paragraphs retrieved are headings or incidental mentions with no real content, since answering would mean guessing past the evidence.

**What guardrails did you implement?**  
Examples: refusal rules, thresholds, limits on snippets, safe defaults.

> A stopword list so common words like "the" or "how" cannot fake a match. A minimum evidence ratio requiring at least half the query's meaningful words to appear in a snippet. A top_k limit of 3 snippets so RAG only ever sees a small, ranked set. A prompt rule requiring the model to answer only from the snippets and use the exact refusal phrase when they are not enough.

## 6. Limitations and Future Improvements

**Current limitations**  
List at least three limitations of your DocuBot system.

1. Whole word matching misses plural forms like "user" vs "users" and splits identifiers like user_id, so real answers can be invisible to the retriever.
2. Paragraph chunking can separate a heading from the content that actually answers the question.
3. Naive mode has no idea what app it is describing, so it answers generic questions confidently instead of admitting it lacks the codebase. It also depends on an external API and fails during an outage.
4. RAG inherits every retrieval mistake. If the retriever misses the right paragraph, RAG refuses even though the docs contain the answer.

**Future improvements**  
List two or three changes that would most improve reliability or usefulness.

1. Add stemming or plural handling so "users" and "user" count as the same word.
2. Chunk by heading section instead of blank line, so a heading and its content stay together.
3. Add a second, looser retrieval pass when the first pass returns nothing, before falling back to a refusal.

## 7. Responsible Use

**Where could this system cause real world harm if used carelessly?**  
Think about wrong answers, missing information, or over trusting the LLM.

> A developer who trusts naive mode could ship code against endpoints or functions that were invented and do not exist in this app. Someone who trusts a RAG refusal as proof the docs are silent could waste time rewriting documentation that already exists, or conclude a feature like the users table is missing when it is not.

**What instructions would you give real developers who want to use DocuBot safely?**  
Write 2 to 4 short bullet points.

- Treat naive LLM answers as generic background reading, not facts about this codebase. Verify anything specific against the actual docs or source.
- Treat a refusal as "the retriever did not find it," not as "it is not in the docs." Try rephrasing the question before assuming the docs are silent.
- Check the cited file yourself before acting on a RAG answer.
